# 实验四　　流水线性能实验

## 实验内容

1. 学习如何使用gem5对程序进行性能分析
2. 评估不同流水线配置对系统整体性能的影响
3. 实际运用Amdahl定律

## 模拟前的问题

1. 如果将程序中的指令分为整数、浮点和内存指令三类，你认为每个类别在程序中所占比例是否相等？

在多数情况下，这三种指令的比例不会相等。

具体到这次的工作负载，对于DAXPYWorkload整数计算几乎集中于循环，内存指令在于读取数据，程序的大部分都是循环和浮点数计算，因此在这个具体的工作负载中，我认为浮点指令和整数指令占比很大，内存指令占比较小；

对于HelloWorkload，不涉及很多浮点计算，内存操作也不会很多，应该是整数计算指令占大多数。

2. 你认为不同的程序是否会有不同的三类指令构成比例？为什么？

对于不同的程序这三类指令的比例当然是不同的。浮点计算是在科学计算，AI学习过程中需要大量使用的计算，在这种程序中浮点计算的比例会很高；在包含大量分支预测，循环语句的程序中，整数计算指令也会很多，用于跳转等分支操作；在数据库管理程序中中会出现频繁的数据存取访问，所以在数据库程序中就会有很多的内存指令。

1. 在这 3 种参数配置组合中，哪一种是你发现的最佳设计组合？

在仿真过程中，可以发现发射延迟为2, 浮点操作延迟为3的时候实现了最佳性能。

2. 为什么你认为在问题 1) 中选择的设计能达到最佳性能？请解释为什么更倾向于优化其中一种延迟而不是另一种？

以第一组延迟配置为参考，可以发现第二个优化了发射延迟，第三组优化了浮点操作延迟。从顺序流水线考虑，第二组减小发射延迟，可以实现不同指令之间的时间间隔变小，很符合顺序流水线拆分工作流程频繁取指令的操作特征，提高了并行度，所以实现了性能的提升。
对于第三组，优化的是浮点操作，虽然可以使得实际计算过程变快，但是由于发射延迟变大，导致指令之间间隔很大，无法充分发挥顺序流水线的并行优势，所以体现出的优化不明显。所以对于顺序流水线，优化发射延迟收益是很大的，所有会更加优先优化发射延迟。

1. 使用 Amdahl 定律和你从第一步收集的信息来预测每种改进情况相对于
   baseline的加速比。你会选择哪种设计？注意：你只能使用步骤Ⅰ收集到的
   数据来回答该问题。

![laws](laws.png)

这个是amdahl定律，alpha代表的是受益于条件改善导致被优化的那一部分时间的比例（代表原始程序中可以被本次优化加速的部分时间占比），k代表了资源改善的任务的加速比（可加速部分的加速比），S是理论加速比。

如果认为延迟减半等于优化部分性能翻倍，对于第二组优化，k=2,alpha=0.46154（这里是上面得出的整数指令占比）\*4/(0.46154\*4 + 0.30769\*8 + 0.23075\*1)=0.40678,计算得出第二组优化的理论加速比是1.25532;

对于第三组优化，k=2, alpha=0.30769\*8/(0.46154\*4 + 0.30769\*8 - 0.23075\*1)=0.54237,计算得出第三组的理论加速比是1.37209;

理论上来讲，优化浮点数操作时间对程序影响更大，性能提升更显著。

2. 每种改进情况相对于baseline的加速比是多少？

通过实际计算第二三组的仿真时间／第一组时间，可以得出实际加速比。这里直接列出一个表进行对比：

| \#  | 理论加速比 | 实际加速比 |
| --- | ---------- | ---------- |
| 1   | 1          | 1          |
| 2   | 1.25532    | 1          |
| 3   | 1.37209    | 1.046748   |

3. 如果你对问题 1 和 2 的回答之间存在差异，你认为可能的原因是什么？

差异主要体现在实际加速比小于理论加速比。可能是因为实际上由于受到内存延迟等各种限制导致实际性能收益k不能简单当作翻倍，实际上的k应该小于2.

其次由于精度问题，在计算指令时间占比的时候三个比例加起来略小于1(0.46154+0.30769+0.23075=0.99998)，都会导致计算与实际结果的偏差。

**_另外，我的实验在docker容器内完成仿真，可能也会造成性能损失导致仿真结果偏离正常结果_**
